# 詩的背景故事 撰寫規範

## 1. 目的與定位

替 `data/poems.js` 中評價最高的前 50 首詩，各撰寫一篇 800～1000 字的「背景故事」，收錄於 `data/poem_story.js`。

現有 `data/author_biography.js` 只回答「這個人是誰」；玩家在花月裡反覆練習的是**某一首詩**，卻不知道「這首詩當下發生了什麼」。「詩的背景故事」補上這一塊：每一首詩背後的具體事件、心境與動機，讓玩家從「背得出來」進化到「懂它為什麼被寫出來」。

這些故事將搭配圖片與語音做成短視頻，同時作為推廣素材——因此文字必須有勾住觀眾的力道，念出來就有畫面。

---

## 2. 核心設計原則

寫作前務必內化以下五條，這是判斷一篇故事「是否合格」的根本標準：

1. **故事性優先**：這是一篇好看的故事，不是詩詞解說，不是知識條目。角色在做事，畫面在推進——不是「這句話的意思是……」的教學。
2. **尊重詩人**：幽默感必須建立在尊重詩人的前提上。人生低谷是「反差」，不是拿來嘲笑的笑點。禁用「廢柴」「魯蛇」「失敗者」等貶低性字眼。
3. **共感而非仰望**：不要把詩人塑造成遙不可及的聖人。呈現他們也會失落、想家、跟朋友喝到爛醉、暗戀、互相嗆聲——真實感才能連結現代玩家。
4. **必備反轉**：每篇要有一個「你以為是這樣，其實是那樣」的認知翻案。翻案內容必須是詩眼的深層命中，不是獵奇噱頭。
5. **回讀動機**：完稿自問——玩家看完，會不會更想把這首詩背熟？若答案不夠肯定，就是失敗，必須重寫。

---

## 3. 敘事語氣規範

**3.1 敘事者身分**：詩人的知心好友。用「跟你說我朋友 XX……」的第一人稱親近語氣，可以吐槽、可以護短、可以幫詩人講出他沒說出口的心事。禁用旁觀者第三人稱冷眼敘述。

**3.2 口語節奏**：句子長短交錯，適時丟一個問句或轉折。文字必須念出來有起伏、能撐配音。禁用書面語堆疊。

**3.3 幽默要淡**：全篇最多 1～2 處輕度幽默，且必須是「處境的自嘲」而非「對詩人的嘲笑」。整篇不能靠段子撐場。

**3.4 絕不文青抒情**：禁用「歲月如梭」「思念如潮水」「淚眼婆娑」「山河依舊」等抒情陳腔。用具體畫面代替感嘆詞。

**3.5 絕不 markdown**：故事全文純散文。禁用粗體、斜體、破折號畫重點、引號當標題、分點列表。想強調的地方，用文字的節奏和句法本身去強調，不靠符號。

---

## 4. 撰寫流程 SOP

每一篇動筆前，依序執行以下五個步驟。不可省略、不可急就章。

### 4.1 找詩眼

通讀全詩後，用一句話回答：

> 這首詩最打動人心的核心情感／人性痛點是什麼？

好的詩眼必須是一個**具體的情境或人性動作**，不能是抽象標籤。

範例對比：

| 詩 | ❌ 標籤式（失敗） | ✅ 具體式（成功） |
|---|---|---|
| 遊子吟 | 母愛 | 當時看不懂的母愛，多年後才突然明白 |
| 琵琶行 | 同病相憐 | 共享同一個黃金時代卻互不相識，多年後淪落成同一形狀 |
| 江雪 | 隱士清高 | 政治犯用不釣魚的釣竿告訴世界「我還在」 |

詩眼寫成標籤，故事就會變成解說；詩眼寫成具體情境，故事才有方向。

### 4.2 開人事時地物清單

正式撰寫前，把這首詩涉及的元素完整列出：

| 維度 | 內容 |
|---|---|
| 人 | 詩中出現的所有人物（含被暗指的角色）、詩人本人、詩人當時身邊的親友 |
| 事 | 詩中描寫的事件、詩人當下的處境事件、以及兩者之間的關聯事件 |
| 時 | 詩寫作的年月、詩中描繪的時間、詩人生平中的重要時間點 |
| 地 | 詩中出現的地名、詩人當時所在的地方、詩人早年生活過的地方 |
| 物 | 詩中出現的具體物品（酒、月、琴、衣、樹、江等）——這是最好的畫面錨點與首尾呼應素材 |

清單開完後，才開始想「哪些元素之間可以連起來？」

### 4.3 主動編織隱藏連接

如果詩人與詩中人物或場景之間有明顯落差、看似無關，必須主動追問：

> 「他們有沒有可能在某個時空曾經同框？共享過同一段回憶？」

案例：〈琵琶行〉琵琶女詩中自述「本是京城女，家在蝦蟆陵下住」，這是長安真實地名；白居易當年也在長安當翰林學士，出入蝦蟆陵教坊聽曲是京城年輕士人的日常。「他當年可能也在後排聽過她的琴」——正史查無記載，但合情合理，就是編故事的關鍵。

案例：〈江雪〉柳宗元被貶永州時，好友劉禹錫同時被貶朗州，兩人隔著幾百里通信。編一句「他知道劉那邊此刻大概也是這樣的雪」，讓「獨」字有具體的呼應對象。

**隱藏連接的三個判準**（三項都必須通過）：

1. **合乎時空**：兩人在該時空必須有共存的可能
2. **合乎身分**：以雙方身分，該場景不突兀
3. **服務詩眼**：這個連接必須加強詩眼的力道，而不只是製造噱頭

三個判準若有一項不成立，該連接必須放棄，不能硬拗。

### 4.4 敘事骨架與金句處理

**四段結構**（每段約 200 字，總長 800～1000 字）：

**第一段：畫面開場（約 200 字）**
從一個具體畫面切入（雪天江邊、油燈下的針線、月光下的旅舍）。開場三句內必須有懸念鉤子——一個讓讀者想繼續看下去的疑問或反差。

常見鉤子類型：
- 識別型：「他手裡那根釣竿三個小時沒動過。」——為什麼？
- 反差型：「這首所有人都會背的詩，其實不是寫在離別的那晚。」——那是什麼時候寫的？
- 疑問型：「跟你說我朋友宗元的一個下午。」——他做了什麼？

**第二段：場景推進（約 300 字）**
帶出詩人當下的處境、事件、人物關係。此段最容易變成生平流水帳——要嚴格圍繞「詩眼」推進，跟詩眼無關的生平資訊全部剪掉。

**第三段：金句時刻（約 200 字）**
詩眼推到最高點的瞬間，金句自然浮現。**金句只能透過三種方式露出**：

- 角色說出口（對白）
- 角色寫下來（動作）
- 敘事者講到那一刻自然帶出（情境）

絕不能單獨切一行出來加破折號解釋。整篇故事中**詩句原文只出現一次**，就是金句時刻這一次。

**第四段：收尾閉環（約 150 字）**
用開場的物件或畫面再次出現，形成回音。可以拉一個時空的長鏡頭——「十年後 XX 死在 XX，沒能等到 XX，但他這個下午留下的畫面活到現在」——讓當下的場景與千年後的讀者產生連結。

**詩句名詞打散原則**：

詩中至少 5～7 個具體名詞（人名、地名、物件、動作）要事先打散進第一、二段的故事情節裡，讓玩家讀完故事、回頭看詩句時，每一個名詞都能對應到剛剛的畫面。這才是「加深記憶點」的實質做法。

例如寫〈江雪〉時，「鳥飛絕」「腳印被雪蓋掉」「只有他一個人」「蓑笠」「釣竿」等意象要先在故事裡出現，最後詩句登場時，讀者的畫面已經在腦中就位。

### 4.5 完稿檢查表

每篇完稿後對照以下項目自檢，任何一項不通過就重寫：

**內容面**
- [ ] 開過「人事時地物」清單
- [ ] 編織了至少一個「隱藏連接」
- [ ] 詩中 5～7 個具體名詞打散進故事情節
- [ ] 金句有明確的「觸發瞬間」，不是敘述完貼標籤引用
- [ ] 開場三句內有懸念鉤子
- [ ] 首尾有物件或畫面的閉環呼應
- [ ] 說中了 4.1 找到的詩眼（不是抽象標籤）
- [ ] 有明確的反轉點

**形式面**
- [ ] 字數落在 800～1000 之間
- [ ] 全篇零 markdown 符號
- [ ] 敘事者是「跟你說我朋友 XX……」的知心好友口吻
- [ ] 詩句原文全篇只出現一次

**禁忌面**
- [ ] 「這句話」「這個字」「這 X 個字」「注意這個」「你看到手法了嗎」等教師口吻句子出現次數 = 0
- [ ] 無貶低詩人的字眼（廢柴、魯蛇、失敗者等）
- [ ] 無文青抒情陳腔（歲月如梭、思念如潮水等）
- [ ] 無「詩句—解釋」逐句拆解格式

**朗讀測試**
- [ ] 念起來像朋友講故事，不像老師劃課文重點

## 5. 絕對禁止事項

以下事項只要犯一項，整篇作廢重寫：

1. **禁用貶低詩人的字眼**：廢柴、魯蛇、失敗者、蠢貨、渣男等
2. **禁用「詩句 + 破折號 + 解釋」的排版格式**（這是解說文的視覺特徵）
3. **禁用教師口吻的句型**：
   - 「這句話的意思是……」
   - 「注意這個字……」
   - 「這 X 個字為什麼這麼重？」
   - 「你看到手法了嗎？」
   - 「你有沒有想過……」（用於引導解讀時）
4. **禁用 markdown 語法**於故事全文：粗體、斜體、破折號畫重點、引號當標題、分點列表
5. **禁止文青抒情**：歲月如梭、思念如潮水、淚眼婆娑、山河依舊、風月無情等
6. **禁止逐句拆解全詩**：故事不是課文分析，全詩原句只在金句時刻出現一次
7. **禁止超過 1000 字**：字數失控代表節奏失控，甜蜜區 800～900 字
8. **禁止把詩眼寫成標籤**：「這首詩表達了母愛／思鄉／懷才不遇」是失敗的詩眼定位
9. **禁止杜撰史實**：編織隱藏連接可以，但重大生平事實不可造假。無考據處要保持故事感又不硬拗——可用「江湖傳說是這樣」的處理，並在敘事語氣中誠實暗示（如「相傳」「據說」）

## 6. 正例與反例對照

### 反例：解說式寫法（〈春曉〉舊版）

> 「春眠不覺曉」——不是睡懶覺爽，是他終於不用一大早起床趕行程、不用趕考、不用去長安低頭做人。這句「不覺曉」的自由感，只有真正放下的人才寫得出來。
> 「處處聞啼鳥」——不是描寫鳥叫，是他終於能聽見自然的聲音，而不是官場的攀比。
> 「夜來風雨聲」——他昨晚睡得很熟，但恍惚聽見了風雨。
> 「花落知多少」——他沒有起來看。他不再急著計較得失。就讓花掉幾朵吧。

**失敗診斷**：
- 詩句被切成四行單獨解釋——這是課文分析的格式
- 主詞是「這句」「這個」，不是「孟浩然」在做什麼
- 沒有場景推進，沒有時間流動
- 讀起來像老師劃重點，不是朋友講故事

### 正例：故事式寫法（〈江雪〉v2 金句段落）

> 雪下得很大。他抬頭看四周，所有的鳥都不見了；他來時踩出的腳印，也被新雪蓋掉了。整條瀟水、整片山巒，此刻只有他一個人。這種安靜大到不像人間。
>
> 他忽然覺得這樣也好。如果整個世界要把他忘掉，如果永貞革新永遠不會被平反，如果他的抱負永遠沒機會兌現，那也沒關係。他還在這裡，他還在做事，他還沒放下釣竿。
>
> 他從懷裡掏出紙筆，寫了二十個字：千山鳥飛絕，萬徑人蹤滅。孤舟蓑笠翁，獨釣寒江雪。

**成功關鍵**：
- 主詞是「他」在做事（抬頭看、覺得、寫下）
- 詩句作為柳宗元寫下的動作，自然出現一次
- 詩中意象（鳥飛絕、腳印被蓋、只有他一個人、蓑笠、釣竿）已經先被故事鋪陳過，讀者看到原詩會自動對應到剛剛的畫面
- 沒有一句「這句話的意思是……」

## 7. 資料結構規範

### 7.1 檔案位置與命名

- 新版寫入：`data/poem_story.js`
- 舊版留存於：`data/poem_story_舊版.js`（比較用，勿修改）

### 7.2 資料格式

以 POEMS 陣列中的 `id` 為 key（比 `title` 穩定，同名詩可能重複）：

```js
const POEM_STORY = {
  277: {
    title: "江雪",
    author: "柳宗元",
    summary: "一句話金句摘要（短影音標題感，帶懸念或反轉）",
    story: `800~1000字背景故事全文，純散文，無 markdown 符號。
使用 template literal 保留換行。`
  },
  ...
};
```

### 7.3 模組匯出

檔案結尾需支援瀏覽器與 Node.js 雙環境：

```js
if (typeof window !== 'undefined') {
  window.POEM_STORY = POEM_STORY;
}
if (typeof module !== 'undefined' && module.exports) {
  module.exports = POEM_STORY;
}
```

### 7.4 段落注釋

每首詩前加一段簡短的作者資訊註解，方便日後審閱：

```js
// --------------------------------------------------------
// id=277 中唐 柳宗元 〈江雪〉rating=6
// 詩眼：政治犯用不釣魚的釣竿告訴世界「我還在」
// 隱藏連接：劉禹錫同時被貶朗州，共享同一片雪
// --------------------------------------------------------
```
