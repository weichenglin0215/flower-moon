# 《花月》隨遇而安 - 企劃書與開發心得

> **定位說明**：本頁不是遊戲，是一頁**無計分、無勝負、無失敗**的純觀看型視覺療癒頁面（沿用 `zhexianren.js` 謫仙人所開創的「舒壓頁」品類，見 `.agent/skills/花月開發常見錯誤與解法.md` 對「遊戲類型」與「資料類型」以外第三種頁面型態的實踐）。因此本企劃書省略遊戲企劃書規範中專屬於「勝負判定」與「難度分級計分」的章節（§難度參數精密控制、§計分系統），改以「物理系統設計」與「互動細節」為核心章節。
>
> **實作後修正紀錄（2026-07-20）**：`suiyuean.js` + `suiyuean.css` 已完成並整合上線，開發過程中依測試回饋多輪修正，詳見 §5、§6、§7 內文標註，摘要如下：
> ① 合成句子由橫排改為**直排**顯示，貼合傳統直書美感；
> ② 碰撞白框由「圓形外接框」改為**貼合字塊實際長寬的長方形**，避免長句子還有段距離就誤判碰撞；
> ③ 碰邊緣與碰撞的「瞬間加速」原本會無限疊加，改為 `_clampSpeed()` 確實封頂；
> ④ 修正碰撞反彈的相對速度方向寫反（`velB-velA` 而非 `velA-velB`）的邏輯 bug，導致真正互相靠近的字反而不會套用反彈衝量；
> ⑤ 每次碰邊緣加入 `WALL_JITTER` 隨機偏轉角度，解決字塊落入完全水平/垂直、永遠撞不到其他字的死路徑；
> ⑥ 有效合併的白框改為套用該句子的顏色，與無效碰撞的白色做出區別；
> ⑦ 取材規則由「單一首詩」改為「多首詩詞混拆湊字數」，解決高字數需求下常態只能選到少數超長詩詞的問題，且保證每首詩至少貢獻偶數句（≥2 句、符合「兩句一聯」原則）；
> ⑧ 新增手指/滑鼠拖曳擾動字塊的互動，並修正 `init()` 初始化順序 bug（`bindEvents()` 誤在 `this.canvas` 賦值前執行，導致拖曳監聽器掛載到 `null` 上完全失效）。

## 第一部分：企劃書 (Design Document)

### 1. 頁面概述與核心目的

**頁面名稱**：隨遇而安 (Free-Floating Poem)
**頁面目標**：從題庫混拆出一批詩句，把每個字拆散、隨機灑落於深色畫面中；玩家點擊「隨遇而安」後，這些字便開始各自漂移、碰撞、依詩句原有的字序重新聚合成句——玩家不需要「做」任何事也能看完整個過程，也可以隨手拖曳畫面主動撥弄字塊的軌跡。
**設計目的**：花月絕大多數頁面都圍繞著「計分」與「勝負」展開，長期高頻遊玩容易產生疲勞感。隨遇而安刻意反其道而行——**沒有分數、沒有時間壓力、沒有失敗結局**，字會不會準時聚合、句子拼成的順序如何，全部交給物理模擬與隨機性決定，玩家的角色從「挑戰者」退回「觀看者」（或想動手時，是「輕輕撥弄的旁觀者」）。這是花月生態中專門對應「不想被考驗，只想安靜看詩」這種心理狀態所設計的功能頁，頁名「隨遇而安」本身就是設計哲學的直接宣告。

### 2. 互動流程與核心操作

#### A. 使用流程

1. **進入頁面** → 系統呼叫 `_pickPoemLines()` 混拆出本輪詩句（見 §5.1），並以 `_buildChunks()` 把每個字拆成一顆靜止的「字塊」，隨機灑落在畫面各處
2. **靜止觀察** → 此時所有字塊速度雖已預先隨機賦值，但物理迴圈尚未啟動（`this.started = false`），畫面呈現「詩句被打散後靜止漂浮」的初始構圖，玩家可以先花時間瀏覽這些字
3. **點擊「隨遇而安」** → `this.started = true`，物理迴圈開始逐幀更新：每個字各自朝隨機方向、隨機速度漂移，略為減速；碰到畫面邊緣會反彈並略微加速、附加隨機偏轉角度
4. **字與字相碰** →
   - 若屬於同一句的相鄰位置：合併成一組，尺寸放大，繼續以合併後的整體移動
   - 若不屬於同一句相鄰位置：彈開，瞬間略微加速
   - 兩種情況都會在碰撞處出現白色（或已成句字塊的專屬色）圓角粗邊框特效與對應音效
5. **句子逐步成形** → 每句字首次合併（湊滿 2 字以上）即依「第幾句率先完成連接」的順序著色（紅、橙、黃…依句數平均分布色相）；句與句彼此永不合併，各自獨立漂移
6. **拖曳互動（隨時可用）** → 玩家可用手指或滑鼠在畫面上拖曳，半徑範圍內的字塊會被主動推開，讓玩家能干預漂移軌跡、加速或延遲某句字的相遇
7. **全部句子拼齊** → 畫面不會停止或彈出任何結算視窗，字塊仍持續漂移、碰撞、彈開；「隨遇而安」按鈕重新浮現
8. **重新開始** → 點擊按鈕即重新混拆一批新詩句、重新灑落，回到步驟 1

**沒有勝利也沒有失敗**：這是本頁與花月其他頁面最根本的差異——步驟 7 之後什麼都不會發生，玩家可以選擇繼續看下去、動手撥弄，或直接點擊重來，一切節奏由玩家自己決定。

#### B. 操作方式

| 平台 | 開始/重來 | 拖曳擾動 |
|------|-----------|---------|
| **桌機（滑鼠）** | 點擊「隨遇而安」按鈕 | 按住滑鼠拖曳畫面 |
| **手機（觸控）** | 輕觸「隨遇而安」按鈕 | 單指拖曳畫面 |

拖曳一律以 Pointer Events（`pointerdown`/`pointermove`/`pointerup`/`pointercancel`/`pointerleave`）實作，同時涵蓋滑鼠與觸控裝置，並以 `setPointerCapture` 確保手指移出畫布範圍時仍能持續追蹤到 `pointerup`。

**手機適配挑戰**：畫布容器已設定 `touch-action: none`，拖曳手勢完全由 JS 接管（`e.preventDefault()`），避免觸控滑動被瀏覽器誤判為頁面捲動。

#### C. 即時反饋機制

- **有效合併**（同句相鄰字相碰）：白色圓角粗邊框變為該句已分配的顏色、線寬較粗（5px）、存續時間較長（360ms）；音效使用 `SoundManager.playGuzheng()` 高八度區間隨機音階，餘韻至少 1 秒漸弱消失
- **無效碰撞**（非相鄰或不同句字相碰）：白色圓角粗邊框、線寬較細（3px）、存續時間短（100ms）；音效使用 `SoundManager.playHit()` 短促音
- **拖曳中**：字塊即時跟隨手指位移方向被推開，越靠近拖曳點推力越強

#### D. 沒有勝負標準

本頁刻意不設計任何勝利或失敗結局——這既是體驗設計，也是核心賣點。全部句子拼齊後畫面不會停頓、不會結算，只會重新顯示按鈕供玩家選擇是否重來。

#### E. 觀看/互動的建議節奏

- **純觀看模式**：點擊開始後完全不動手，單純欣賞字塊隨機碰撞聚散的過程，體驗接近水族箱或熔岩燈的療癒感
- **主動撥弄模式**：用拖曳手勢刻意把兩個相鄰的字推向彼此，加速某一句的成形；或反過來把即將相撞的無關字分開，延長「混沌」的觀賞時間

### 3. 介面設計與使用者體驗

#### A. 主介面佈局

```
┌─────────────────────────────────────┐
│ 隨遇而安                          ✕  │  ← 左上角低調標題／右上角關閉鍵
│                                       │
│      字      字            字        │
│                                       │
│   字      字      字   字            │  ← 深色 Canvas 全螢幕畫布，
│                                       │     字塊隨機灑落/漂移其上
│      字            字      字        │
│                                       │
│              [隨遇而安]              │  ← 按鈕：初始與全部完成後顯示
└─────────────────────────────────────┘
```

- 整頁為單一 `<canvas>` 全螢幕繪製，不使用 DOM 節點呈現每個字（效能考量：單輪最多近百個獨立移動物件，Canvas 繪製優於逐一操作 DOM）
- 左上角標題、右上角關閉鍵為浮動 DOM 元素，疊加於畫布之上

#### B. 視覺回饋系統

| 狀態 | 視覺表現 |
|------|---------|
| 尚未合併的單字 | 白色（`hsl(0,0%,96%)`），帶淡白色發光 |
| 已合併成句的字塊 | 依該句色相著色（`hsl(hue,78%,66%)`），帶同色發光 |
| 有效碰撞瞬間 | 該句專屬色的粗圓角框，線寬 5px，360ms 內淡出並略微放大 |
| 無效碰撞瞬間 | 白色圓角框，線寬 3px，100ms 內快速淡出 |

色相分布公式：`hue = 總句數 <= 1 ? 0 : (完成合併的第 N 句 × 300 / (總句數-1))`，讓紅（0°）到紫（300°）的色彩沿完成順序均勻鋪開，避免落在色環尾端與起點的紅紫混淆。

#### C. 背景設計

深藍黑徑向漸層（`hsl(220,35%,14%)` 中心 → `hsl(0,0%,2%)` 邊緣），提供足夠暗度讓彩色成句字塊與白色單字在視覺上清楚跳出。

#### D. 使用體驗的流暢度設計

- **拖曳即時回饋**：即使尚未點擊「隨遇而安」開始漂移，拖曳仍會立即位移字塊（見 §5.4），確保玩家在任何狀態下操作都「有感」，不會誤以為互動失效
- **速度上限與碰撞加速並存**：碰邊緣、碰撞、拖曳都會為字塊注入額外速度，但統一經 `_clampSpeed()` 封頂，避免長時間運行後速度無限疊加導致畫面失控
- **邊緣防卡死設計**：`WALL_JITTER` 讓每次反彈都帶一點隨機偏轉角度，字塊不會困在完全水平或垂直的軌跡上、永遠遇不到其他字

### 4. 機制的繼承與創新

**繼承**：物理碰撞、彈開、封頂速度等基礎手法承襲自花月既有的 Canvas 類遊戲慣例（如 game5～game7 的移動判定），色彩分配邏輯則呼應 `zhexianren.js`「依進度決定視覺呈現」的設計精神。

**創新設計決策**：

**⓵ 「合併即語意判定」的物理碰撞**：一般碰撞遊戲的合併規則多半只看「同色」或「同形」，本頁的合併條件是「同一句的相鄰字位置」——碰撞偵測本身直接承載了語文知識（字序），而非單純的視覺屬性比對。這讓物理模擬與詩詞學習自然結合，而不是把詩詞當作物理引擎的皮膚。

**⓶ 「多詩混拆湊字數」取代「單一首詩硬湊」**：初版企劃要求單一首詩滿足 56～98 字的字數目標，但多數詩詞長度不足，導致實測時畫面反覆只能抽到極少數超長詩（見 §5.1 修正說明）。改為「每次抽一小段固定偶數句、逐段累加不同首詩」後，畫面能穩定達到目標字數，且視覺上呈現多首詩交織漂移的效果，比單一首詩更有「萬千詩句同游」的意境。

**⓷ 「靜止時也能拖曳」的互動設計**：多數物理模擬類頁面只有在動畫啟動後才回應互動，本頁刻意讓拖曳在字塊靜止階段就能生效（直接位移字塊位置），確保玩家一進入頁面就能立即感受到「這是一個可以互動的畫面」，而不是要先點擊按鈕才「解鎖」互動能力。

**⓸ 「零失敗結局」的物理系統永續運行**：全部句子拼齊後物理迴圈不會停止，字塊會無限期地繼續碰撞漂移下去，直到玩家主動點擊重來。這與傳統遊戲「達成目標即結束」的慣性完全相反，呼應頁面「隨遇而安」的核心命題——沒有終點，只有持續流動的當下。

### 5. 核心物理與視覺設計

#### 5.1 詩詞取材：多首詩混拆湊字數（重大修正）

初版直接呼叫一次 `getSharedRandomPoem(minRating, 8, 12, 56, 98, ...)`，要求單一首詩同時滿足 8～12 句、56～98 字——但符合這種長度的詩詞在題庫中占比極低，導致實測畫面反覆只能挑到少數幾首超長詩，字數與句數的多樣性嚴重不足。

修正後改為迴圈累加式取材：

```javascript
while (totalChars < TOTAL_MIN_CHARS && poemCount < MAX_POEMS && attempts < MAX_PULL_ATTEMPTS) {
    const remaining = TOTAL_MAX_CHARS - totalChars;
    if (remaining < PULL_MIN_LINES * 2) break; // 剩餘空間連最短一聯都放不下，收尾

    const result = getSharedRandomPoem(
        POEM_MIN_RATING, PULL_MIN_LINES, PULL_MAX_LINES,
        PULL_MIN_LINES * 2, remaining, "", null, 'suiyuean'
    );
    // 保險：確保偶數句，若意外拿到奇數句就捨去最後一句
    let pulled = result.lines;
    if (pulled.length % 2 !== 0) pulled = pulled.slice(0, pulled.length - 1);
    if (pulled.length < 2) continue;

    const pulledChars = pulled.join('').length;
    if (totalChars + pulledChars > TOTAL_MAX_CHARS) continue;

    lines = lines.concat(pulled);
    totalChars += pulledChars;
    poemCount++;
}
```

每次只向題庫抽一小段（`PULL_MIN_LINES=2` 到 `PULL_MAX_LINES=8` 句，且強制偶數），逐段累加、最多混合 `MAX_POEMS=10` 首不同的詩，直到總字數落在 `[TOTAL_MIN_CHARS, TOTAL_MAX_CHARS] = [56, 98]`。這個設計同時滿足了「畫面字數要夠多」與「絕不出現單句/奇數句破壞兩句一聯原則」兩個看似衝突的需求。

#### 5.2 碰撞判定：AABB 長方形取代外接圓

字塊合併後會直排延伸（見 §5.5），若用外接圓做碰撞判定，長句子的圓半徑會遠大於實際字塊寬度，導致「明明畫面上還有一大段距離，卻已經觸發碰撞」的失真體驗。修正後改用緊貼字塊實際尺寸的長方形（AABB）：

```javascript
const overlapX = (ma.halfW + mb.halfW) - Math.abs(dx);
const overlapY = (ma.halfH + mb.halfH) - Math.abs(dy);
if (overlapX <= 0 || overlapY <= 0) continue; // 未重疊

// 取穿透較淺的軸作為碰撞法線（MTV，最小平移向量）
if (overlapX < overlapY) { nx = dx >= 0 ? 1 : -1; ny = 0; overlap = overlapX; }
else { nx = 0; ny = dy >= 0 ? 1 : -1; overlap = overlapY; }
```

法線方向取「穿透較淺的軸」（Minimum Translation Vector），讓反彈方向真正對應碰撞發生的相對位置，而非單純以中心連線方向計算。

#### 5.3 碰撞反彈：修正相對速度方向寫反的 bug

早期版本將相對速度寫成 `velA - velB`，導致「兩者正在互相靠近」時計算出的 `velAlongNormal` 反而落在程式判定為「正在分離」的分支，衝量完全不會套用——結果只有位置被推開，速度方向卻毫無變化，玩家會看到「兩個字明明撞在一起，卻只是貼著滑開，完全沒有真正反彈」的失真手感。修正為標準的 `velB - velA`：

```javascript
const relVx = b.vx - a.vx, relVy = b.vy - a.vy;
const velAlongNormal = relVx * nx + relVy * ny;
if (velAlongNormal < 0) { // 真正互相靠近時才觸發反彈
    const j = -velAlongNormal * 2 / (1 / massA + 1 / massB);
    // ...施加衝量、瞬間加速...
}
```

修正後才真正符合「兩字塊相碰，會依碰撞角度分解法線/切線分量各自反射」的物理直覺。

#### 5.4 拖曳擾動：位置直推 + 速度延續的混合設計

```javascript
_disturbChunks: function (wx, wy, dx, dy) {
    // dx/dy 先夾限最大位移，避免手指瞬移造成字塊爆衝
    const r2 = DRAG_RADIUS * DRAG_RADIUS;
    for (const c of this.chunks) {
        const d2 = (c.x-wx)**2 + (c.y-wy)**2;
        if (d2 > r2) continue;
        const f = 1 - Math.sqrt(d2) / DRAG_RADIUS; // 越靠近拖曳點，推力越大

        c.x += dx * f * DRAG_POS_STRENGTH; // 直接位移：靜止時也有立即回饋
        c.y += dy * f * DRAG_POS_STRENGTH;
        c.vx += dx * f * DRAG_VEL_STRENGTH; // 附加速度：漂移中會有延續動能
        c.vy += dy * f * DRAG_VEL_STRENGTH;
        this._clampSpeed(c);

        // 拖曳可能把字推出畫面，直接夾回邊界（未啟動漂移時牆壁反彈邏輯不會執行）
        c.x = Math.max(m.halfW, Math.min(STAGE_W - m.halfW, c.x));
        c.y = Math.max(m.halfH, Math.min(STAGE_H - m.halfH, c.y));
    }
}
```

同時採用「直接位移」與「附加速度」兩種手法而非二選一：純粹加速度的做法在字塊靜止（未點擊開始）時完全無感（因為物理迴圈根本不會執行 `_integrate`）；純粹位移的做法則在漂移中會顯得「瞬移感」過重、缺乏物理連續性。兩者混合後，靜止狀態立即有反應，漂移狀態則保有動能延續的自然感。

開發時發現一項初始化順序 bug：`init()` 原本在 `this.canvas` 賦值**之前**就呼叫了 `bindEvents()`，導致拖曳監聽器綁定時 `cv = this.canvas` 拿到的是尚未賦值的 `null`，`cv.addEventListener(...)` 直接拋出例外、整段拖曳功能形同虛設卻無任何畫面異狀（因為例外發生在初始化流程中，不影響其餘功能運作）。修正為先取得全部 DOM 參照，再呼叫 `bindEvents()`。

#### 5.5 合成句子的直排顯示

```javascript
_drawChunk: function (ctx, c) {
    const spacing = c.fontSize * 1.05;
    const totalHeight = spacing * (c.chars.length - 1);
    const startY = c.y - totalHeight / 2;
    for (let k = 0; k < c.chars.length; k++) {
        ctx.fillText(c.chars[k], c.x, startY + spacing * k); // x 不變，y 逐字往下
    }
}
```

已合併的句子由上而下直排顯示（單字則單獨置中，不受影響），貼合中文傳統直書美感，也讓長句子的碰撞外框呈現直立長條形而非橫向長條，與「字塊逐字往下疊」的直覺一致。

### 6. 音樂與音效設計

- **背景音樂**：無持續播放的背景樂，保持純粹的碰撞音效節奏，避免額外旋律干擾「安靜看詩」的療癒基調
- **操作音效**：
  - 有效合併（同句相鄰字相碰）：`SoundManager.playGuzheng(idx, 0.85)`，索引隨機落在高八度區間（8～14），音階較高、餘韻長達 1～2.5 秒漸弱消失，象徵「一句話又拼齊了一點」的滿足感
  - 無效碰撞：`SoundManager.playHit(idx, 0.1)`，索引隨機落在 0～4，音效極短促，不打斷觀看節奏
- **正確/錯誤回饋音的等價轉換**：本頁沒有「答對/答錯」的概念，但沿用花月「有意義的動作給予較長較高的音效、無意義的動作給予短促音效」的一貫聽覺語彙，讓玩家即使不看畫面，光憑聲音節奏也能感受到「這次是不是拼成一句了」
- **聲音與視覺的協同**：碰撞瞬間的白框特效與音效同幀觸發，兩者持續時間也刻意對應（有效合併框較大較久、音效較長；無效碰撞框較小較快、音效極短），確保視聽節奏完全一致

## 第二部分：開發心得與分析

### 1. 頁面性分析：「這是一場沒有終點、只有持續流動的當下」

隨遇而安在花月的所有頁面中是一個異數——它不要求玩家做出任何判斷，甚至完全不做也能獲得完整體驗。玩家的心理狀態不是「準備迎接挑戰」，而是「放下評判，單純觀看」；即使動手拖曳，那也是一種「輕輕撥弄」而非「精準操作」的互動姿態。

這種設計對應的不是提取練習效應，而是花月生態中少見的「被動暴露學習」——玩家在毫無壓力的狀態下，反覆看見同樣的字被拆散又重組，句子完成瞬間的顏色躍動與音效自然形成記憶錨點，即使玩家從未「主動背誦」，這些詩句的字形與聲韻仍在潛移默化間留下痕跡。

**頂點時刻**：不是某個困難關卡的通關瞬間，而是當畫面上多首詩詞同時進行到不同完成度——有的句子已經成紅色整句安靜地在角落漂浮，有的還只是零散的白色單字彼此追逐——玩家隨手一撥，兩個原本毫不相干的字被推向彼此，恰好完成了另一句的最後一次合併，色彩瞬間躍動、音符悠揚落下。那種「我什麼都沒『刻意』做，卻好像參與了這場拼湊」的鬆弛感，是花月所有頁面中獨一無二的體驗類型。

### 2. 精華推廣亮點

- **「螢幕保護程式」等級的長時間背景播放素材**：本頁沒有結束條件，非常適合作為活動攤位、教室投影或社群直播的背景畫面持續播放，字句不斷聚散帶來的視覺流動感，比靜態海報更能吸引路人目光停留。
  標題可以下：**「這不是遊戲，是一首會自己拼回原樣的詩——花月最療癒的角落」**

- **「拖曳撥弄」的解壓互動話題**：拖曳讓原本不相干的字碰在一起、看它們彈開或合併的過程本身極具紓壓感，類似戳泡泡紙的心理機制，適合作為「上班/唸書中場休息」的短影音素材。
  標題可以下：**「累的時候別滑手機了，來撥弄幾個漂浮的詩句字吧」**

### 3. 對外推廣短語提案

> *「沒有分數、沒有時間、沒有輸贏——這裡只有字，慢慢找到彼此。」*

> *「你以為只是看著字漂來漂去？其實每一次碰撞，都是系統在悄悄檢查『這兩個字該不該是鄰居』。」*

> *「多首詩詞混在同一片深藍夜色裡漂流，紅橙黃色依序亮起——那是詩句正在找回自己原本的模樣。」*

### 4. 總結

《隨遇而安》是花月生態中第一款完全捨棄計分與勝負的純觀看型頁面，用「碰撞即語意判定」的物理設計，把詩詞的字序知識悄悄藏進看似隨機的漂移碰撞之中。技術上，從「多詩混拆湊字數」解決題庫長度不足的問題，到 AABB 長方形碰撞、相對速度方向的物理修正，再到拖曳擾動的位置/速度混合設計，每一處修正都是為了讓這場「看似隨意」的漂移盡可能貼近真實、耐看、可互動。這不是一款用來被挑戰的頁面，而是一處讓人願意停下來、什麼都不做也能安靜欣賞的角落——恰如其名，隨遇而安。
