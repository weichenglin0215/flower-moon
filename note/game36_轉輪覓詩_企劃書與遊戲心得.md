# 《花月》Game36: 轉輪覓詩 - 遊戲企劃書與遊戲心得

> **美術風格說明**：本作視覺**完全沿用 Game1（慢思快選）的米白色宣紙風格**，套用 `theme_xuanzhi.css` 既有的 `--fm-*` 共用變數體系（`--fm-paper` 米白紙色背景、`--fm-paper-card` 卡片底、`--fm-ink` 墨色文字、`--fm-cinnabar` 硃紅、`--fm-celadon` 青瓷綠、`--fm-gold` 金、`--fm-border` 米黃邊框），不自創新的深色/星空系配色。所有卡片、按鈕、邊框、陰影皆比照 game1.css 現有樣式直接複用，僅新增 `game36-` 前綴的版面配置 class，色彩全數繼承共用主題。

## 第一部分：遊戲企劃書 (Game Design Document)

### 0. 原型遊戲分析

#### A. 原型遊戲玩法解析

**Wordle**：玩家在六次機會內猜出一個隱藏的目標單詞。每次提交一個等長的猜測後，系統以三種顏色標示每個字母的命中狀態——綠色（字母正確且位置正確）、黃色（字母存在於目標中但位置錯誤）、灰色（字母完全不存在於目標中）。核心爽點在於「漸進式邏輯推理」：玩家每猜一次，都能從顏色回饋中排除或鎖定更多資訊，六次機會構成一條清晰的「資訊增益曲線」，最後一次成功命中時的興奮感，來自於「親手用邏輯一步步逼近真相」的智力成就感。操作方式：桌機鍵盤逐字輸入，行動裝置以虛擬鍵盤點擊輸入。

#### B. 原型的介面設計與吸引力分析

Wordle 的介面設計極簡卻高度資訊密集：六行五格的網格，每一格在提交瞬間翻轉並染色，翻牌動畫（flip reveal）搭配色彩設定，讓「答案揭曉」的瞬間充滿儀式感。這種「網格逐格點亮」的視覺回饋，正是玩家每天願意回來玩一次的核心黏著力來源——顏色本身就是敘事，玩家不需要文字說明就能立刻理解自己的猜測「差多遠」。

花月版本保留了：逐格翻轉染色動畫、三色回饋邏輯，並將「自由鍵盤輸入單詞」置換為「從候選詩句轉輪中選取整句」；同時**捨棄了原版 Wordle「六次機會用盡即失敗」的硬性次數限制**，改為可無限次持續作答的捲動式判定網格（詳見 §2、§4），這是本作在玩法轉譯上最關鍵的兩步。

#### C. 詩詞融合分析：從原型到花月的轉化邏輯

| 問題 | 答案 |
|------|------|
| **「什麼被替換成詩詞」** | 原始 Wordle 中「玩家鍵盤輸入的五字母單詞」被替換成「玩家從轉輪中選取的一句五言或七言詩句」；顏色判定邏輯原封不動保留，只是判定單位從「字母」變成「漢字」 |
| **如何加入語文知識門檻** | 答案不是自由輸入，而是從轉輪候選池中「選」出來的；候選池中除了少量「混淆句」外，**每一句候選詩句都必定與目標詩句共享至少一個字**（複用 game21／game23 已驗證的「依目標字元反查詩句」機制），玩家可以憑肉眼辨識池中哪些字看起來眼熟，再透過顏色回饋確認具體位置——這讓「認字」與「邏輯推理」兩種能力同時派上用場，缺一不可 |
| **詩詞學習效果** | 強化「字序記憶」與「語感記憶」的複合體——玩家透過反覆猜測與顏色回饋反推目標詩句每個字的正確位置，這種「以整句為單位進行逐字校對」的過程，比單純背誦更能鞏固詩句內部的精確字序；由於沒有次數上限，玩家可以放心地把每一次猜測都當成一次純粹的推理練習，而非患得患失地計算「機會還剩幾次」 |

#### D. 手機觸控適配：矛盾點與解決方案

**矛盾一：垂直轉輪拖曳與頁面捲動手勢衝突**
- **原始操作**：Wordle 原生無捲動手勢問題（鍵盤輸入）
- **觸控矛盾**：花月版本的候選詩句轉輪需要垂直拖曳瀏覽，這與瀏覽器原生的「上下滑動捲頁」手勢完全相同，容易觸發瀏覽器捲動而非轉輪滾動
- **解決方案**：轉輪容器套用 CSS `touch-action: none`，並在 `touchmove` 事件中呼叫 `event.preventDefault()` 完全接管垂直手勢，由遊戲自行計算位移量並映射為轉輪旋轉角度

**矛盾二：選取「正中央」項目的精度問題**
- **原始操作**：桌機可用滑鼠精確點擊任一候選項
- **觸控矛盾**：手指觸控時，若轉輪仍在慣性滑動中，很難精確點擊「當下剛好在正中央」的項目，容易點到相鄰的候選句
- **解決方案**：中央判定區域擴大為固定的「命中窗口」（約佔轉輪高度 20%），任何完全落在此窗口內的候選句都視為「當前選取項」；點擊事件只在轉輪完全靜止（無慣性滑動）時才生效，避免滑動中誤觸提交錯誤答案

**矛盾三：翻牌染色動畫與轉輪操作的時序衝突**
- **觸控矛盾**：若玩家在上一次提交的翻牌動畫尚未播放完畢時，就急著拖曳轉輪準備下一次猜測，容易造成動畫與輸入狀態不同步
- **解決方案**：提交後鎖定轉輪的 `pointer-events`，直到該列翻牌動畫（每格間隔 0.15 秒，共 N 格、約 N×0.15 秒）完整播放結束才解鎖，確保「回饋看完才能繼續」的節奏感，同時避免狀態機混亂

### 1. 遊戲概述與核心目的

**遊戲名稱**：轉輪覓詩 (Verse Reel / Poetry Wordle)
**遊戲目標**：透過上下拖曳「詩句轉輪」選取候選詩句作為猜測，依據系統回饋的三色提示逐步逼近，**不限次數**地持續作答，直到猜中隱藏的目標詩句為止；上方判定網格可垂直捲動，隨猜測次數增加不斷向下延伸新的一列。
**學習目的**：相較於花月其他遊戲多半是「認出正確字/句」的單次判斷，轉輪覓詩要求玩家進行「跨回合的邏輯推理累積」——每一次猜測後的顏色回饋都是一條新線索，玩家必須主動比對「這句詩的哪些字、哪些位置」與目標相符，這是一種將**字序記憶**與**演繹推理**深度結合的全新提取方式。對熟悉目標詩句的玩家而言，這是一場「秒殺式的知識驗證」；對僅有模糊印象的玩家而言，這是一場「用文學直覺輔助邏輯推理」的漸進式回憶挑戰——兩種玩家都能在遊戲中獲得成長，且無論猜了多少次，最終都能親手抵達答案，而非半途被系統宣告失敗。

### 2. 遊戲玩法與核心操作

#### A. 遊戲流程

1. **選擇難度** → 系統依難度設定選取一句評分達標的五言或七言詩句作為目標（玩家不可見），同步生成該局的候選詩句轉輪池
2. **開局呈現** → 畫面上方顯示**可垂直捲動的判定網格**（初始顯示 6 列，每列格數＝目標詩句字數），畫面下方顯示可上下捲動的詩句轉輪，正中央項目為當前選取候選
3. **捲動選句** → 玩家上下拖曳轉輪瀏覽候選詩句，轉輪具備慣性滑動效果，鬆手後逐漸減速停止
4. **提交猜測** → 點擊正中央的候選詩句，確認採用該句作為本次猜測，該句字元填入判定網格的下一個空行
5. **顏色判定** → 網格該列以逐格翻轉動畫（每格間隔 0.15 秒）依序翻開染色：
   - **青瓷綠**（`--fm-correct-*`）：該字存在且位置與目標詩句完全相同
   - **提示藍**（`--fm-hint-*`）：該字存在於目標詩句中，但位置不同
   - **淡墨灰**（`--fm-ink-faint` + `--fm-paper-deep`）：該字完全不存在於目標詩句中
6. **網格自動延伸** → 若判定網格已填滿當前可視的 6 列，第 7 次猜測會自動在網格底部新增第 7 列（第 8、9…列以此類推），網格區域轉為可上下捲動，讓玩家隨時回顧先前所有猜測列的顏色線索
7. **持續猜測** → 重複步驟 3-6，**沒有次數上限**；已猜過的候選句在轉輪中標記為半透明勾選態，方便玩家避免重複選取（但不會從轉輪中移除）。由於系統保證候選池中「非混淆句都含有目標字」（見 §5.2），且已猜過的候選不能重複提交，玩家只要有耐心逐一排除，理論上必定能在候選池耗盡前找到正確答案
8. **勝利條件**：任一列的所有格全數翻為青瓷綠（猜中目標詩句），立即結束本局並觸發勝利結算
9. **沒有失敗條件** → 本作**不會**中途強制結束或直接公布正確答案；玩家可以持續作答直到自己猜中為止，也可以隨時主動點擊「放棄本局」離開（見 §2-D 詳細說明）

#### B. 操作方式

| 平台 | 捲動轉輪 | 選取候選 | 提交猜測 |
|------|---------|---------|---------|
| **桌機（滑鼠）** | `mousedown`+拖曳，或滾輪 `wheel` | 候選句停留於正中央窗口即為選取狀態 | `click` 正中央候選句 |
| **手機（觸控）** | 單指垂直拖曳（`touchmove`，含慣性滑動） | 同上（正中央命中窗口判定） | 單指輕觸正中央候選句 |

**操作限制**：
- 只有完全落入「正中央命中窗口」（轉輪高度約 20% 範圍）的候選句才可被點擊提交，其餘候選句點擊無效，避免誤觸
- 轉輪仍在慣性滑動中時，點擊事件暫時停用，需等待轉輪完全靜止
- 翻牌染色動畫播放期間（約 1.2 秒）鎖定轉輪互動，確保回饋節奏完整

**手機適配挑戰**：垂直拖曳轉輪與瀏覽器原生捲頁手勢的衝突是最大挑戰，透過 `touch-action: none` 與 `preventDefault()` 完全接管手勢（詳見 §0-D 矛盾一）。

#### C. 即時反饋機制

- **轉輪捲動中**：候選句依與中心的距離呈現「2D 仿 3D 轉軸」視覺——越接近中心字體越大越清晰（不透明度 100%），越遠離中心則垂直壓縮（`scaleY` 縮小）且不透明度遞減，模擬滾筒透視效果
- **候選句進入命中窗口**：該句底色浮現淡金色高亮框，提示「現在點擊即可提交這句」
- **提交瞬間**：播放 `playConfirmItem()` 音效，該候選句短暫閃爍後，其文字「飛入」頂部網格對應列
- **翻牌染色**：每格翻轉時搭配清脆的翻牌音效，翻至背面瞬間依判定結果播放對應色彩的微光特效（綠色微光／黃色微光／灰色無光）
- **整列判定完成**：若該列出現至少一個綠字，播放 `playSuccessShort()`；若整列全為淡墨灰（完全不相關），播放低音量的 `playFailure()` 提示「這次猜測沒有交集」（僅是單次提示音，不代表遊戲結束）；若整列全綠（猜中），觸發勝利流程

#### D. 勝利標準（本作沒有失敗標準）

**勝利**：任一次猜測使該列所有格全數翻為青瓷綠，立即觸發 `ScoreManager.playWinAnimation()` 勝利結算流程，並顯示目標詩句完整資訊（含詩名、作者、朝代）。

**本作刻意不設計「失敗」機制**：這是與原版 Wordle 最根本的差異之一。真人版 Wordle 只要打開鍵盤就能自由輸入下一個猜測，重來的成本近乎於零；但花月版本的猜測是「從候選池中挑選」，若比照原版設定「六次用盡即失敗、公布答案、重新開局」，玩家等同於必須：重新記住剛才已經排除的所有線索、重新在轉輪裡把候選句捲過一遍、甚至因為系統換了新的目標句而讓先前的推理全部歸零——這種「打掉重練」的挫折成本，遠高於傳統 Wordle 直接重新打字的成本，會讓玩家在卡關時直接放棄，而非繼續挑戰。因此本作改為：**判定網格可無限向下捲動延伸，猜測次數沒有上限，玩家永遠可以繼續嘗試，直到自己找到正確答案為止**。若玩家中途不想再繼續這一局，可隨時點擊左上角「放棄本局」按鈕主動離開，但系統本身絕不會強制結束或代為公布答案。

#### E. 獲得高分的技巧

- **善用首猜的資訊價值**：第一次猜測應優先選擇轉輪中「字元組合豐富、跨主題」的候選句，即使猜錯，也能透過大量黃/灰回饋快速排除或鎖定多個字，比盲猜相似句更有效率
- **善用已確定的綠字定位**：只要某位置已翻綠，後續猜測應優先從轉輪中尋找「該位置字元相同」的候選句，加速收斂
- **留意黃字暗示的移位線索**：黃色字提示「這個字在答案中，但不在這裡」，可用來排除轉輪中「相同位置仍放該字」的候選句，聚焦真正符合條件的選項
- **熟悉詩句可直接秒殺**：若玩家一眼認出轉輪中某句正是自己熟悉的名句，可以跳過邏輯推理直接提交，這是知識儲備直接轉化為遊戲效率的最短路徑

### 3. 介面設計與使用者體驗

#### A. 主介面佈局（由上到下）

```
│ ❤️❤️❤️❤️❤️❤️                     │
├─────────────────────────────────────┤
│  ↕ 判定網格（可垂直捲動）              │
│  ┌───┬───┬───┬───┬───┐            │
│  │ 綠 │ 藍 │ 灰 │ 灰 │ 綠 │  ← 第1列（已猜）
│  ├───┼───┼───┼───┼───┤            │
│  │ 灰 │ 綠 │ 藍 │ 灰 │ 綠 │  ← 第2列（已猜）
│  ├───┼───┼───┼───┼───┤            │
│  │ 灰 │ 綠 │ 綠 │ 藍 │ 灰 │  ← 第3列（已猜）
│  ├───┼───┼───┼───┼───┤            │
│  │  （持續下捲可見第4、5、6…列）  │
│  └───┴───┴───┴───┴───┘            │

│              …… 上一句候選（縮小、淡出）      │
│         ┌─────────────────────┐    │

│         └─────────────────────┘    │
│              …… 下一句候選（縮小、淡出）      │
├─────────────────────────────────────┤
│         （上下拖曳轉輪／點擊中央提交）         │  ← 操作提示
└─────────────────────────────────────┘
```

- **頂部資訊列**：分數、已嘗試次數（純資訊顯示，不設上限）
- **判定網格區**：可垂直捲動，初始顯示視窗約 3-4 列，隨猜測次數增加自動向下延伸新的一列（第 7、8、9…列），每列格數＝目標詩句字數（5 或 7），已猜列翻牌染色定格；網格右側或底部提供輕量捲軸提示，讓玩家可隨時上滑回顧先前所有猜測列
- **詩句轉輪區**：佔畫面下半部，2D 仿 3D 垂直捲軸，中央候選句最大最清晰，上下候選句依距離縮小淡出

#### B. 視覺回饋系統

本作**完全套用 game1（慢思快選）的米白宣紙視覺語言**，不引入任何新色系；所有狀態顏色皆直接取自 `theme_xuanzhi.css` 既有的 `--fm-*` 變數，僅在「字存在但位置錯誤」這個 Wordle 特有的第三態上，從既有變數中挑選最貼近「線索、尚未確定」語感的既有色階，而非發明新色。

| 狀態 | 視覺表現 | 色彩規格（沿用 game1 既有變數） |
|------|---------|---------|
| 網格空格（未猜） | 米黃色細邊框方格，宣紙底色透出 | `border: 2px solid var(--fm-border, #d8c193)` |
| 判定：位置正確 | 方格底色填滿，墨色粗體字 | `var(--fm-correct-bg)` 底 + `var(--fm-correct-border)` 邊 + `var(--fm-correct-text)` 字（青瓷綠，與 game1 答對格完全同色） |
| 判定：字存在但位置錯誤 | 方格底色填滿，墨色粗體字 | `var(--fm-hint-bg)` 底 + `var(--fm-hint-border)` 邊 + `var(--fm-hint-text)` 字（沿用 game1 的提示藍，代表「有線索但未確定」，與其他遊戲的「提示」語意一致） |
| 判定：完全不存在 | 方格底色填滿，淡墨色字 | 卡片底 `var(--fm-paper-deep)` + 邊框 `var(--fm-border)` + 字色 `var(--fm-ink-faint)`（宣紙自然的淡墨灰調，不使用生硬的純灰色） |
| 轉輪中央候選（命中窗口） | 字體最大（40px），宣紙卡片浮起，金色高亮外框 | 卡片 `var(--fm-paper-card)` + 邊框 `var(--fm-border-strong)` + `box-shadow: var(--fm-shadow-soft)` |
| 轉輪鄰近候選 | 字體中等（28px），墨色字，透明度略降 | `var(--fm-ink-soft)`，不透明度 70% |
| 轉輪遠端候選 | `scaleY(0.5)` 垂直壓縮，透明度更低 | `var(--fm-ink-faint)`，不透明度 35% |
| 已選過候選（歷史標記） | 半透明硃紅勾銷符號疊加於該句右側 | `var(--fm-cinnabar)`，不影響底色 |

**配色核心準則（比照 game1 既有規範，不新增色系）**：
- 青瓷綠（`--fm-correct-*`）＝位置正確：與 game1 答對時的格子同色，玩家跨遊戲的色彩直覺完全一致，不需要重新學習「這個綠色代表什麼」
- 提示藍（`--fm-hint-*`）＝存在但位置錯誤：借用 game1 既有的「提示」色階而非發明新的黃色系，語意上恰好吻合——「有價值的線索，但還沒完全確定」正是 game1 的 hint 語意
- 淡墨灰（`--fm-ink-faint` + `--fm-paper-deep`）＝完全不存在：不使用花月慣用的硃紅錯誤色，因為 Wordle 式玩法中「猜錯」是推理過程的常態而非失誤；宣紙自帶的淡墨灰調，比外來的灰階色塊更融入整體美術風格

#### C. 計時器呈現方式

計時器採用**矩形邊框 SVG**形式，環繞整個判定網格區的四邊，以 `stroke-dashoffset` 動畫表現剩餘時間。**計時器只影響最終得分，不會中止遊戲**——時間歸零後不再累積時間獎勵分數，但玩家仍可無限次數繼續作答直到猜中為止；這個設計確保玩家不會因為「快沒時間了」而倉促亂猜，時間壓力只用來拉開熟練玩家與新手玩家的分數差距，而非製造強制性的失敗結局。

#### D. 使用體驗的流暢度設計

- **命中窗口容錯設計**：正中央可提交區域刻意設計為轉輪高度 20% 的寬容範圍，而非要求像素級精確停止，大幅降低觸控裝置的操作挫折感
- **已選標記防重複**：轉輪不會移除已選過的候選句（移除會造成其餘項目跳動、破壞轉輪視覺連續性），改以半透明勾選標記提醒，玩家仍可自由捲回查看
- **翻牌節奏的儀式感**：逐格翻轉搭配 0.15 秒間隔，讓玩家有時間逐格消化顏色訊息，而非一次性同時翻開造成資訊過載
- **網格自動捲動至最新列**：每次新增一列判定結果後，網格自動平滑捲動至該列完全可見，玩家不需要手動找到「剛剛猜的是哪一列」
- **零挫折感的核心設計**：本作徹底移除「失敗」這個結局——沒有次數上限、沒有強制公布答案、沒有懲罰性視覺（無紅色警示、無扣分特效），玩家唯一會經歷的負面情緒是「這次猜測沒有交集」的短暫提示音，而非「你輸了，重新開始」的挫敗宣告；這讓玩家可以完全專注於推理本身，而不必分心計算「還剩幾次機會」

### 4. 遊戲機制的繼承與創新

**繼承**：本作繼承 Wordle 的核心猜謎迴圈——逐格顏色回饋、資訊漸進收斂的邏輯推理過程，這是近年來最成功的文字解謎範式之一，玩家對其規則的學習成本極低；但刻意**不繼承**原版「固定六次機會、次數用盡即失敗」的硬性規則（詳見下方⓸）。

**創新設計決策**：

**⓵ 「選詩句」取代「打字母」：解決中文輸入與答案唯一性的雙重難題**：英文 Wordle 可以讓玩家自由鍵盤輸入任意五字母組合（不需要是「有意義的單詞」也能提交，系統只驗證是否為合法單詞）。但中文詩句若開放自由輸入，會產生龐大的合法性驗證問題（玩家可能拼湊出根本不存在於任何詩詞中的漢字組合）。花月版本改用「轉輪選句」機制，讓玩家永遠只能從系統預先篩選過的、真實存在於詩詞資料庫中的候選句提交，從根本上解決了合法性驗證問題，同時把「選擇」本身變成了一種需要文學知識輔助判斷的操作。

**⓶ 「2D 仿 3D 轉軸」視覺化候選池，把選擇過程變成一場探索**：候選池不是傳統的下拉選單或列表，而是模擬實體滾筒選字機的視覺呈現——候選句隨距離中心遠近縮放淡出，玩家「轉動」轉輪尋找答案的過程本身帶有探索的儀式感，而非枯燥的清單捲動。

**⓷ 色彩判定邏輯完全借用 game1 既有語彙，零新增色系**：本作刻意不為 Wordle 的第三種判定狀態（存在但位置錯誤）發明新顏色，而是直接借用 game1 既有的「提示藍」（`--fm-hint-*`）——因為這個狀態的語意本質就是「有價值的線索，但還沒完全確定」，與 game1 的提示語意完全吻合，玩家不需要學習任何新的色彩詞彙，就能把跨遊戲累積的色彩直覺無縫沿用到本作。

**⓸ 移除次數上限，把「重來的代價」降到零**：原版 Wordle 的「六次機會」設計，建立在「重新開局的成本極低」（打開鍵盤重新輸入即可）這個前提上；但花月版本的猜測必須從轉輪候選池中挑選，若沿用固定次數上限，玩家在次數用盡後必須：記住剛才辛苦累積的所有顏色線索、換一首新的目標詩句重新推理，過程繁複且挫折感遠高於原版。因此本作將判定網格改為**可垂直捲動、隨猜測次數自動延伸新列**的設計，讓玩家永遠有機會用「剛剛學到的線索」繼續逼近答案，而不會被系統中途打斷、強迫從零開始。這個決策讓「猜錯」徹底從「失敗的代價」轉化為「純粹的資訊」——猜錯的每一列都只是多一條線索，不會有任何懲罰。

### 5. 答案句庫生成的精密設計

這是本作企劃階段爭議最大、也最核心的技術決策——**候選詩句轉輪池該如何組成**，直接決定了這款遊戲究竟是「考驗詩詞知識」還是「純邏輯猜字遊戲」。本節的生成邏輯**直接複用 [game21.js](../game21.js)（橫批成詩）與 [game23.js](../game23.js) 已經過實戰驗證的「依目標字元反查詩句」機制**，不重新發明演算法。

#### 5.1 核心限制：候選句必須與目標句「等長」

Wordle 色彩判定的前提是「逐字元位置比對」，因此**候選池中每一句用於提交猜測的候選詩句，字數都必須與目標詩句完全相同**（5 言配 5 言、7 言配 7 言），這是不可妥協的技術硬限制。系統延伸 `script.js` 既有的 `_buildPoemLineIndex()`（`window.getSharedBarLine` 內部已建立的 char→詩句索引）加上長度過濾，取得「句長＝N」且「含有特定字元」的候選清單，避免每局遊戲都重新掃描全部詩句。

#### 5.2 核心規則修正：候選句（混淆句除外）必須至少包含目標句中的一個字

這是本節最重要的設計原則，也是與初版企劃書最大的差異之處：

> **除了刻意設計的「混淆句」之外，轉輪中出現的每一句候選詩句，都必須與目標詩句「床前明月光」共享至少一個字**（例如候選句中含有「床」、「前」、「明」、「月」、「光」其中任一字皆可）。

**原因**：若候選池中充斥著與目標句毫無關聯的句子，玩家將完全無法「用肉眼閱讀」來輔助判斷——轉輪會退化成一堆隨機漢字組合，玩家只能盲目亂猜等待顏色回饋，這違背了花月「靠語文知識勝出，不靠運氣」的根本設計哲學（見 [遊戲企劃書撰寫規範.md](../.agent/skills/遊戲企劃書撰寫規範.md) §1.1）。透過保證「非混淆句必定與目標字共享至少一字」，玩家可以：
1. 憑閱讀直覺快速掃過轉輪，鎖定「看起來眼熟／含有目標詩句用字」的候選句，大幅縮小瀏覽範圍
2. 即使不記得目標詩句的精確字序，也能透過「這句含有『月』字」的視覺線索，主動選擇它來測試「月」字究竟落在哪個位置
3. 顏色回饋因此永遠「有跡可循」——黃色格永遠對應到玩家肉眼可辨識的那個共享字，形成閱讀能力與邏輯推理的雙重驗證

#### 5.3 候選句生成演算法：逐字元反查法（複用 game21/23 邏輯）

演算法核心概念與 game21 的 `prepareChallenge()` 完全一致：**針對目標詩句的每一個字，各自呼叫一次 `getSharedBarLine()` 取得含有該字的其他詩句**，只是額外加上「句長＝目標句長度」的過濾條件（game21 的橫批玩法允許 4-12 字變動長度，本作因需要逐格對齊，必須鎖定為等長）。

```javascript
// 以目標句「床前明月光」（5字）為例
function generateCandidatePool(targetLine, poolSize, decoyRatio) {
    const targetChars = targetLine.split('');   // ['床','前','明','月','光']
    const targetLen = targetChars.length;
    const usedLines = new Set([targetLine]);
    const pool = [targetLine];                  // 池中固定含 1 句正確答案

    // 1. 逐字元反查：每個目標字至少產生 1 句候選，額外配額依 poolSize 動態加碼
    const nonDecoyQuota = Math.round(poolSize * (1 - decoyRatio)) - 1; // 扣掉正確答案本身
    let perCharQuota = Math.max(1, Math.floor(nonDecoyQuota / targetLen));

    targetChars.forEach(ch => {
        for (let i = 0; i < perCharQuota; i++) {
            // 複用 script.js 既有的 getSharedBarLine，額外加上「等長」過濾
            const line = window.getSharedBarLine(ch, {
                minLen: targetLen, maxLen: targetLen,   // ⚠️ 鎖定等長（與 game21 的 4-12 變動長度不同）
                excludePoemId: targetPoem.id,
                excludeLines: usedLines,
                preferMid: true
            });
            if (line) { pool.push(line); usedLines.add(line); }
        }
    });

    // 2. 混淆句：複用 game21 的 pickDecoyLine 邏輯，但同樣加上「等長」過濾，
    //    確保混淆句不含目標句任何一個字
    const decoyCount = poolSize - pool.length;
    const targetCharSet = new Set(targetChars);
    for (let i = 0; i < decoyCount; i++) {
        const decoyLine = pickDecoyLineExactLen(targetCharSet, usedLines, targetLen);
        if (decoyLine) { pool.push(decoyLine); usedLines.add(decoyLine); }
    }

    return shuffle(pool);
}
```

這個實作幾乎是 game21 `prepareChallenge()` 中「為每個謎題字配對直棒」與「干擾棒選句」兩段邏輯的直接移植，僅將原本允許的「4-12 字變動長度」收斂為「與目標句嚴格等長」，因為橫批玩法（game21）靠橫向滑動對齊，而 Wordle 玩法（本作）靠逐格位置比對，兩者都需要「候選句含目標字」的核心保證，只是等長與否的技術要求不同。

#### 5.4 候選句要多少句最適合？

延續 game21/23 已驗證的候選池規模概念，並考量轉輪捲動的可瀏覽性，訂定候選池總數依難度分級如下：

| 難度 | 候選池總數 | 每字保底候選數 | 理由 |
|------|-----------|--------------|------|
| 小學 | 16 句 | 每字約 2-3 句 | 池子小、含目標字候選句比例高，玩家憑閱讀直覺就能快速鎖定範圍，適合入門者建立信心 |
| 中學 | 20 句 | 每字約 2 句 | 略增加池子總量，仍維持每個目標字都有充分候選可供比對 |
| 高中 | 24 句 | 每字約 1-2 句 | 池子接近「無法憑感覺全部瀏覽完」的臨界點，開始需要系統性運用顏色回饋 |
| 大學 | 28 句 | 每字約 1-2 句 | 池子夠大，轉輪捲動距離明顯增加，混淆句比例同步提高（見 §5.5） |
| 研究所 | 32 句 | 每字約 1 句 | 池子最大，且目標詩句改採較冷門的低評分詩句（見 §7），單純知識辨識的優勢大幅降低 |

**設計依據**：16-32 句的區間，換算成轉輪視覺呈現（每次可見約 5-7 句），玩家平均需要拖曳 3-5 次手勢即可瀏覽完整池子；由於本作每一句非混淆候選都保證與目標句有視覺關聯（見 §5.2），瀏覽負擔比「隨機拼湊池」更輕，因此池子規模刻意比一般猜謎遊戲的候選清單再收斂一些，避免「肉眼可辨識的線索」被過度稀釋在龐大池子中。

#### 5.5 要不要有完全不符合的混淆句？比例是多少？

**結論：一定要有，且比例應隨難度遞增，從小學的約 15% 到研究所的約 45%——但無論難度高低，混淆句都只是候選池中的「少數派」，多數候選句仍保證與目標句共享至少一字（見 §5.2 核心規則）。**

原因有三層：

1. **技術面：全灰回饋是重要的推理基準**。如果候選池中每一句都與目標句有共享字（依 §5.2 保證），玩家將永遠無法透過「全灰」直接排除整組字元的可能性，推理效率會大幅降低。適量的混淆句（與目標句零重疊），才能讓玩家在猜測後獲得「這些字統統不對」的乾淨資訊，這是 Wordle 機制運作良好的必要條件，也是 game21 的 `pickDecoyLine`（干擾棒：不含任何謎題字）存在的原因。

2. **設計哲學面：混淆句依然是真實詩句，不是隨機廢字**。混淆句透過 `pickDecoyLine` 邏輯生成，依然是**真實存在的詩詞資料庫中的合法詩句**，只是與目標句零字元重疊。玩家若不認識，會被顏色回饋正確引導排除；玩家若剛好認識這句詩，也是一次額外的、免費的詩詞曝光機會，符合花月一貫「干擾也要有文學價值」的堅持。

3. **難度曲線面：混淆句比例，直接控制候選池的「視覺純度」**。小學難度池中僅 15% 為混淆句，代表轉輪中絕大多數候選句都與目標句有共享字，玩家能快速憑閱讀鎖定範圍；研究所難度池中混淆句比例拉高至 45%，玩家需要花更多心力分辨「哪些候選句才是真正含有目標字的」，同時每個字的保底候選數降到最低（每字約 1 句），大幅壓縮「靠肉眼掃描」的優勢，這正是本作「質變點」的核心之一（詳見 §7）。

**混淆句比例對照表（依難度）**：

| 難度 | 混淆句比例 | 含目標字候選句比例 |
|------|-----------|-------------------|
| 小學 | 15% | 85% |
| 中學 | 20% | 80% |
| 高中 | 30% | 70% |
| 大學 | 38% | 62% |
| 研究所 | 45% | 55% |

**設計註記**：即使在研究所難度，混淆句比例也刻意不超過 50%，確保「含目標字候選句」永遠是池中主流——這是花月版 Wordle 與英文原版 Wordle 最本質的差異：英文 Wordle 的猜測詞彼此毫無關聯純靠邏輯，而花月版本刻意讓多數候選句都與目標句存在字面連結，讓「讀詩的能力」與「邏輯推理」兩者缺一不可，而非退化成單純的排列組合遊戲。

#### 5.6 顏色判定演算法（含重複字元的正確處理）

顏色判定必須正確處理「目標句中含重複字元」的邊界情況（Wordle 系列遊戲最容易出錯的技術細節）：

```javascript
function judgeGuess(guessChars, targetChars) {
    const result = new Array(guessChars.length).fill('gray');
    const targetRemain = [...targetChars]; // 複製一份用於扣除已比對的字

    // 第一輪：先標記所有「位置正確」的綠色，並從 targetRemain 扣除
    for (let i = 0; i < guessChars.length; i++) {
        if (guessChars[i] === targetChars[i]) {
            result[i] = 'green';
            targetRemain[i] = null; // 標記已用掉
        }
    }
    // 第二輪：剩餘位置比對「字存在但位置不同」的黃色，
    // 每次命中就從 targetRemain 移除一個，避免同一字被重複標記黃色
    for (let i = 0; i < guessChars.length; i++) {
        if (result[i] === 'green') continue;
        const idx = targetRemain.indexOf(guessChars[i]);
        if (idx !== -1) {
            result[i] = 'yellow';
            targetRemain[idx] = null;
        }
    }
    return result;
}
```

這個兩輪判定法確保「目標句中只出現一次的字，候選句猜中兩次時，只有一個位置會被標黃，另一個維持灰色」，避免傳統誤植的「重複字都染色」錯誤，這是本作技術實作中最容易被忽略卻最重要的正確性細節。

### 6. 音樂與音效設計

轉輪覓詩採用「靜心推理 + 節奏分明的翻牌回饋」聲音策略，強化「這是一場思考遊戲」而非「反應遊戲」的體驗基調。

**背景音樂**：遊戲進行中播放低音量、規律而沉穩的古箏撥絃循環，節奏刻意放慢，模擬「棋局思考」的氛圍，不使用任何催促感強烈的旋律，讓玩家在轉輪捲動與邏輯推理時保持專注不受干擾。

**操作音效**：
- 轉輪捲動：極輕微的「齒輪喀嗒」聲隨捲動速度變化頻率，模擬實體滾筒選字機的觸感回饋
- 候選句進入命中窗口：短促清脆的「叮」聲，提示「這句現在可以提交」

**正確/錯誤回饋音**：
- 提交猜測：`playConfirmItem()`——宣告「這次猜測已送出」
- 逐格翻牌：每格翻轉搭配細微的翻牌音效，音量隨翻牌速度遞減營造層次感
- 整列出現綠字：`playSuccessShort()`——上揚音效，鼓勵玩家「有進展了」
- 整列全灰：`playFailure()`低音量版本——不使用強烈的失敗音效，僅以低沉單音提示「這次猜測沒有交集」，避免產生過度負面的心理暗示

**特殊音效**：
- 猜中目標（該列全綠）：`playJoyfulTriple()` 搭配 `melodyPlayer.playFullMelody('小星星')` 完整旋律片段，慶祝邏輯推理的最終勝利
- 判定網格新增一列（第 7、8、9…列）：極輕微的「翻頁」音效，提示玩家「網格延伸了，這不是懲罰，只是繼續」
- 計時器歸零（不影響繼續作答）：一聲清亮的鐘聲提示「時間獎勵已結束」，不搭配任何負面音效，因為遊戲並未結束

**勝利結算音**：觸發 `ScoreManager.playWinAnimation()` 標準流程，播放完整慶祝旋律，與其他花月遊戲的勝利聲音語彙保持一致性。

**聲音與視覺的協同**：翻牌動畫的逐格音效與畫面染色完全同步（每 0.15 秒一格），確保玩家透過聽覺節奏就能預判下一格即將翻開，聽覺與視覺共同構成資訊接收的雙重通道。

### 7. 難度參數的精密控制 (Difficulty Scaling)

| 難度 | 詩句字數 | 詩評下限 | 候選池總數 | 混淆句比例 | 時間獎勵窗口 | 提示模式 |
|------|---------|---------|-----------|-----------|------|---------|
| 小學 | 5言 | ★★★★★★ | 16句 | 15% | 240秒 | 顯示目標詩之詩名+作者 |
| 中學 | 5言 | ★★★★★ | 20句 | 20% | 210秒 | 顯示目標詩之朝代 |
| 高中 | 7言 | ★★★★★ | 24句 | 30% | 200秒 | 無提示 |
| 大學 | 7言 | ★★★★ | 28句 | 38% | 180秒 | 無提示 |
| 研究所 | 7言 | ★★★ | 32句 | 45% | 150秒 | 無提示 |

> **關於「時間獎勵窗口」欄位**：本作沒有時間耗盡即失敗的機制（見 §2-D）。此欄位僅代表「時間分數獎勵」的有效窗口——在此時限內猜中，可獲得完整的時間加成分數；超過時限後玩家仍可繼續無限次作答，只是不再累積時間加成，最終分數會略低，但依然能順利完成本局並看到勝利結算。

**質變點在高中至大學之間**：字數從 5 言切換至 7 言，判定網格從 5 欄擴增為 7 欄，候選句的排列組合空間呈指數級成長，同一次猜測所能獲得的資訊量比例相對下降，玩家必須更精準地運用每一次猜測才能維持在時間獎勵窗口內完成。

**研究所的極限設計**：候選池的混淆句比例衝高至 45%（詳見 §5.5），加上每個目標字的保底候選數降到最低（每字約 1 句），意味著轉輪中「憑閱讀直覺快速掃過」的優勢大幅降低；同時詩評下限降至 ★★★，目標詩句不再侷限於膾炙人口的名篇，玩家過去累積的「名句直覺」在此優勢銳減，這是專為頂尖玩家保留的、真正考驗系統性邏輯推理能力的挑戰——但即使如此，玩家依然保有「不限次數、直到猜中為止」的安全網，難度只反映在最終分數高低，而非能否完成這一局。

**特殊視覺/音效差異**：研究所難度的轉輪捲動慣性衰減係數降低（滑動後停止得更慢），模擬「候選項目又多又滑」的失控感，是研究所難度獨有的操作質變。

## 第二部分：遊戲心得與分析 (Game Experience & Analysis)

### 1. 遊戲性分析：「這是一場在滾筒與色彩之間，用邏輯一步步逼近古人心聲的推理遊戲」

《轉輪覓詩》是花月生態中第一款真正意義上的「多回合累積推理」遊戲——過去所有遊戲的每一次判斷幾乎都是獨立事件（答對或答錯，立即見分曉），唯獨這款遊戲要求玩家把一連串猜測視為一個連續的偵探過程，前面的線索必須被記住、被整合，才能在後面的猜測中發揮效用；而且這個過程**沒有次數上限**，玩家可以安心地把每一次猜測都當成單純的資訊蒐集，不必計算「機會還剩幾次」。

這種機制對學習的價值，在於它強迫玩家從「被動辨認」升級為「主動假設與驗證」的高階認知模式。玩家在第一次猜測後看到「這個字位置對了，那個字存在但位置錯了」，大腦必須立刻重組心智模型，思考「這句詩到底可能長什麼樣子」，這種持續性的假設修正過程，正是心理學上「主動建構式學習」的具體實踐，遠比單純的「認出正確答案」更能鍛鍊深層的字序記憶。

從使用體驗的角度，小學難度的候選池充滿與目標詩句相關的句子，玩家幾乎能憑印象快速鎖定範圍，體驗接近「認字尋寶」的溫馨感；隨著難度提升，候選池中混淆句的比例節節攀升，玩家逐漸體會到「認識這首詩」已經不夠用了，必須真正掌握顏色回饋的邏輯規則。到了研究所難度，冷門詩句配上高達四成五的混淆候選，玩家幾乎是在「零先驗知識」的狀態下，純靠反覆猜測累積的資訊推理出答案——但無論猜了多少次，判定網格都會耐心地向下延伸，等待玩家找到那唯一正確的一句。

**頂點時刻**：當玩家在研究所難度下，憑藉前面十幾次猜測累積的青瓷綠、提示藍、淡墨灰三色線索，在第三十二句（候選池最後一句）候選詩句組成的轉輪裡精準捲動停在唯一符合所有已知條件的那一句，點擊提交的瞬間，整列同時翻轉為青瓷綠、全螢幕金光炸裂——那種「即使花了很久，也絕不被系統打斷，純靠邏輯推理走到終點」的智力快感與踏實感，是花月所有遊戲中獨一無二的體驗類型。

### 2. 精華推廣亮點

- **「花月版 Wordle」——天然的每日話題與社群傳播素材**：Wordle 本身已是全球現象級的社群傳播案例（玩家自發分享三色方塊組成的「戰績截圖」而不洩漏答案），花月版本可以直接複製這套「零劇透分享文化」——玩家分享捲動的多列色塊戰績圖，吸引好友猜測「這句詩到底是什麼、你猜了幾次」，形成低成本高效益的自然傳播鏈。
  標題可以下：**「全網最不會逼死人的 Wordle：猜到你猜中為止，破解李白藏在轉輪裡的那一句詩」**

- **「轉輪裡的心理戰」——實況主的猶豫時刻最有戲**：轉輪捲動尋找候選句的過程，天然適合實況互動——實況主會邊捲邊碎念「這句應該不對……等等這個字位置怪怪的……」，觀眾可以在彈幕搶答，形成主播與觀眾共同推理的即時互動場景，而且不用擔心「機會用完了」掃興收場。
  標題可以下：**「不限次數、三十二句候選，我們一起在花月轉輪裡破解這句藏頭詩！」**

### 3. 對外推廣短語提案

> *「你以為這只是猜字遊戲？試試看在三十二句真假難辨的候選詩句裡，不限次數找出唯一的正確答案。」*

> *「小心，藍色不代表你錯了——那是系統在告訴你『這個字在，只是站錯位置』。真正的高手，靠的不是運氣，是邏輯。」*

> *「我們把 Wordle 的靈魂裝進了一座詩句轉輪：每一句候選，都保證與題目詩句共享至少一個字，這不是隨機拼湊的猜謎，這是一場複用花月既有詩句反查引擎、精密設計的文學偵探遊戲。」*

### 4. 總結

《轉輪覓詩》將 Wordle 這個全球現象級的猜謎機制，透過「等長句比對」與「逐字元反查候選池」的精密設計，完整轉譯進花月的詩詞宇宙，同時解決了中文自由輸入的合法性難題。候選池生成演算法直接複用 game21、game23 已實戰驗證的「依目標字元反查詩句」邏輯，確保絕大多數候選句都與目標句共享至少一字，讓玩家的閱讀直覺與邏輯推理缺一不可，僅少量混淆句比例隨難度精密調校；米白宣紙的視覺語言完全沿用 game1 既有配色，不增加玩家的視覺學習成本，讓玩家把全副心力留給推理本身。2D 仿 3D 轉軸的視覺化選句機制，讓「選擇候選句」這個原本可能枯燥的操作，變成一場充滿探索感的滾筒尋寶。這不是一款簡單移植的猜謎遊戲，這是一場用色彩、機率與詩詞資料庫共同編織而成的邏輯推理盛宴。
