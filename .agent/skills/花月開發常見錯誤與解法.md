# 花月開發常見錯誤與解法

本文件根據《花月》的歷史對話與開發紀錄，統計並分析了開發期間常遇到的錯誤型態、對應的最佳解法，以及**模組化與優化**的實作經驗，以提升後續開發效率與防錯能力。

## 1. 遊戲類型 (日曆、Cards、Game 1 ~ Game 15)

### 1.1 邏輯錯誤 (Logic Errors)
- **事件與動畫重複觸發 (Multiple Triggers/Animations)**
  - **錯誤情境**：例如 Game 9 中的通關動畫重複播放，或 Game 5 重生事件連續觸發多次。
  - **解決方式**：統一實作狀態標記（例如 `isAnimating = true` 或 `isCompleted = true`），確保在同一執行週期或狀態下，任何觸發函式只會被執行並播放動畫一次。
- **過關或題目生成數量不配平**
  - **錯誤情境**：Game 4 產生的答案按鈕 (ans-btn) 數量不足以應付該局配置的題目字數，例如 "word count insufficient" 錯誤。
  - **解決方式**：需動態偵測並生成對應數量的按鈕，並於生成後加上完整性驗證 (Validation Check)；確保按鈕陣列數量與可用選項大於等於所需答案長度。
- **隨機演算法分佈品質不佳**
  - **錯誤情境**：Game 8 迷宮路徑生成過於貼近邊緣、容易產生大片空白區塊，或提早過度轉彎導致難度與美觀度不佳。
  - **解決方式**：路徑生成演算法需加入「網格平均分配權重」、「限制過早轉彎次數」及「偏好在特定直線長度後才轉向」的限制條件即可生成出更具挑戰性與美觀的關卡軌跡。另，針對單次挑戰必須鎖定初始起點以確保公平。
- **重置/重生狀態殘留 (State Reset Failure)**
  - **錯誤情境**：Game 5 死亡後未正確延遲回到起點、Game 7 (Flappy Poetry) 特效狀態在重啟遊戲後仍殘留在畫面上。
  - **解決方式**：建立標準化且封裝完整的 `resetGame()` 重置函式，嚴格初始化各物件位置、狀態、倒數計時器及切實執行 `clearTimeout` 等函式清除所有過期的計時任務。

- **⚠️ 遊戲時間限制使用固定設定值而非實際題目字數**
  - **錯誤情境（2026-05）**：`difficultySettings` 中改用 `timeLimitRate` 後，各遊戲的計時公式卻仍以固定設定參數（如 `questionCount`、`answerLen`、`lineCount`、`maxMaskCount`）作為乘數，而非實際執行期間產出的題目字數。例如 `settings.questionCount * settings.timeLimitRate` 或 `settings.lineCount * settings.timeLimitRate`，當詩詞實際字數與設定值不符時，時間限制會不準確。
  - **正確做法**：`timeLimitRate` 必須乘以**執行期間實際產出的題目字數**，即在題目生成（`getSharedRandomPoem` / `selectPoem` / `generateProblem` 等）完成後，使用已填入的陣列/字串長度：
    ```javascript
    // ✅ 正確範例（game15 canonical pattern）：
    // 先呼叫取得詩詞的函式，讓 targetChars 被填入實際字元
    // 然後用 targetChars.length（而非設定值）計算時間
    const calcTimeLimit = this.targetChars.length * settings.timeLimitRate;
    this.timeLimit = calcTimeLimit;
    this.timer = calcTimeLimit;
    
    // ❌ 錯誤反例：使用固定設定值
    // this.timeLimit = settings.questionCount * settings.timeLimitRate;
    // this.timeLimit = settings.answerLen * settings.timeLimitRate;
    // this.timeLimit = settings.lineCount * settings.timeLimitRate;
    ```
  - **各遊戲對應的實際字數來源**：

    | 遊戲 | 實際字數來源 | 呼叫時機 |
    |------|------------|---------|
    | game2 | `(answerLine === 1 ? line1 : line2).replace(punct, '').length` | `selectPoem()` 後 |
    | game4 | `this.hiddenPositions.length` | `selectRandomPoem()` 後 |
    | game5 | `this.targetChars.length` | `preparePoem()` 後 |
    | game6 | `this.enemies.length` | `loadLevel()` 後（`startTimer()` 內） |
    | game7 | `this.poemChars.length` | `loadPoem()` 後 |
    | game8 | `this.fullPoemText.length` | `selectRandomPoem()` 後 |
    | game13 | `hiddenMeta.filter(m=>m.isHidden).length + hiddenChars.length` | `generateProblem()` 後 |
    | game15 | `this.targetChars.length` | `getSharedRandomPoem()` 後 |

  - **根本原則**：計時初始化必須放在詩詞/題目生成函式**之後**，絕不能在之前。若 `retryGame()` 不重新生成題目，則使用上一局留下的陣列/字串長度即可，不必重新呼叫取詩函式。

### 1.2 介面錯誤 (Interface Errors)
- **UI 元素自動關閉與焦點遺失**
  - **錯誤情境**：遊戲初始卡片 (Intro Card / AboutDialog) 因計時器過短自動消失，玩家來不及閱讀。
  - **解決方式**：涉及重要說明指引的 UI，應取消定時自動關閉，改為強迫玩家手動觸發（Explicit interaction，如點擊、滑動或按下確認鈕）才隱藏卡片，以提升使用者體驗。
- **顯示資訊排版擁擠**
  - **錯誤情境**：Game 7 的全詩結算視窗無法顯示完整內容、過關 Block 高度不符文字。
  - **解決方式**：透過允許 Scrolling (滾動條) 及依據螢幕比例放大 MessageBox，來容納完整篇幅。
- **⚠️ CSS 全域污染：跨模組 class 名稱衝突導致介面異常**
  - **錯誤情境（2026-03）**：日曆介面（`calendar.js`）的詩句第一行在 PC 與手機瀏覽器上都往上偏移約 60px。排查後發現 `cards.css` 中定義了 `.poem-line:first-child { transform: translateY(-60px); }` 規則，而 `calendar.js` 渲染詩句時也使用了相同的 `.poem-line` class 名稱。由於 CSS 是全域生效的，只要兩個頁面同時載入兩份 CSS，任何重名 class 都會互相干擾。
  - **解決方式**：
    1. 將衝突模組（`cards.css`）的 class 名稱加上模組前綴，改為 `.cards-poem-line`，並同步更新 `cards.js` 中的 class 賦值。
    2. 在所有的 JS 模組（`calendar.js`、`cards.js` 等）頂部或 `init()` 中加入 `loadCSS` 防護函式（詳見「3.3 CSS 載入防護機制」）。
  - **根本預防原則**：**每個模組的 CSS class 名稱必須加上模組前綴**（如 `cards-`、`calendar-`、`game1-` 等），絕不使用通用名稱（如 `.poem-line`、`.title`）作為特定模組的私有 class。

### 1.3 PC / 手機瀏覽器介面差異 (Cross-Device/RWD Issues)
- **碰撞範圍在不同畫面尺寸的偏差**
  - **錯誤情境**：手機窄螢幕上的碰撞範圍 (Hitbox) 計算因比例壓縮而與直覺產生偏誤。
  - **解決方式**：繪圖時需將絕對座標轉化為相對於容器的比例值 (如 `%` )；邏輯判斷時應帶入當前的縮放係數 (Scale factor) 全局即時調整碰撞檢測邊界。

## 2. 資料類型 (上述遊戲之外的核心邏輯，如 ScoreManager、Menu、成就等)

### 2.1 邏輯錯誤 (Logic Errors)
- **單點功能新增但配套系統遺漏**
  - **錯誤情境**：開發完新遊戲 (Game 5 或 Game 6) 後，發現分數未被記錄，或無法在成就大廳中選擇對應關卡。
  - **解決方式**：建立新增遊戲的 SOP 清單。在增加新模組後，必須同步去檢查並註冊 `scoreManager.js` 分數池追蹤、修改 `menu.js` 的捷徑按鍵，以及串接成就系統中的遊戲解鎖驗證判斷式。
- **分數防弊與存取異常**
  - **錯誤情境**：多遊戲併發存檔時可能有 LocalStorage 被複寫的問題。
  - **解決方式**：資料存取統一透過建立好、具容錯防護與解密能力的 `ScoreManager` 共用介接，避免散落各處單獨呼叫 `localStorage.getItem` 或 `setItem`。

### 2.2 介面錯誤 (Interface Errors)
- **全域特效坐標定位失敗 (例如飛星動畫)**
  - **錯誤情境**：`createFlyingStar` 在獲取目標元素（如總分表）的座標點時，偏移量設置粗糙導致特效錯位亂飛。
  - **解決方式**：DOM 元素定位時不使用單純字串轉數字的 `px` 絕對相加，應透過 `getBoundingClientRect()` 獲取精準位置，並考慮目前畫布的縮放比例（`window.stageScale`）來推算換算比，精確實作跨容器的動畫軌跡。

### 2.3 PC / 手機瀏覽器介面差異 (Cross-Device/RWD Issues)
- **SafeArea 與滾動條卡頓問題**
  - **錯誤情境**：首頁選單或資料統計清單在 iOS Safari 上被底部返回鍵或系統 SafeArea 遮擋；原生滾動存在卡頓感。
  - **解決方式**：在 CSS 設定上，確保有滾動需求的層疊區域加上 `-webkit-overflow-scrolling: touch;`；並強制利用 CSS 變數 `padding-bottom: env(safe-area-inset-bottom)` 增加動態底距，確保下層項目能被拉升至安全區內。

## 3. 共用優化機制與整合實踐 (Shared Modules & Optimizations)

在《花月》的持續開發迭代中，為了解決高重複度發生的各種排版與設備問題，我們實作並抽象出了**共用的基礎建設核心模組**，這是最重要的一次重構：

### 3.1 視窗適配系統整合 (Screen Adaptive System: `screen_adaptive.js` / `screen_adaptive.css`)
- **歷史痛點 (Pain Points)**：
  早期開發中的 `game1-overlay` ~ `game9-overlay` 寬高多為獨立攥寫或固定 `px`，每當使用不同長寬比的設備（如 iPad、PC 寬螢幕、iPhone 細長螢幕）就會發生版面被切、字體過大覆蓋介面，或是內容未滿全螢幕等破圖現象。且 `rem` 單位容易受系統字體縮放影響，導致排版跑位。
- **優化實作方案 (Implementation)**：
  我們整合了**全容器等比例縮放系統**，由 CSS 定義邏輯舞台，搭配 JS 動態計算 `transform: scale()`，其優化細節包含：
  1. **固定邏輯尺寸 (500x850 px)**：
     - 在 `screen_adaptive.css` 中定義 `#stage` 容器的固定尺寸為 500x850px。所有子元件均使用 `px` 作為單位，確保在邏輯空間內的絕對穩定。
  2. **視窗智慧縮放 (`applyScale` 方法)**：
     - 在 `screen_adaptive.js` 中，即時偵測 `visualViewport` 或 `window.innerWidth/Height`。
     - 縮放公式：`scale = Math.min(vw / 500, vh / 850)`。取較小值以確保舞台內容永遠完整顯示在視窗內，不超出邊界。
  3. **絕對 PX 佈局 (Absolute Pixel Layout)**：
     - 廢棄 `rem` 單位，改用純 `px`。因為舞台被整體 `scale()` 縮放，內部的 `px` 會跟著等比例縮小或放大，且不受瀏覽器系統字體大小設定的干擾。
  4. **座標導航與防抖 (Coordinate Mapping & Debounce)**：
     - 暴露 `window.stageScale` 與 `window.stageRect` 供外部模組（如飛星特效、浮動視窗）進行座標轉換。
     - 監聽 `resize` 與 `visualViewport` 事件，確保在手機鍵盤彈出或旋轉螢幕時能即時重新對齊。
- **結論**：
  在往後的所有網頁、關卡、對話框開發中，只需將內容放入 `#stage` 容器並以 `px` 作為單位基準，即可享有無痛跨設備適配的最佳體驗，根絕過去因螢幕比例或系統字體導致的排版錯誤。

### 3.2 詩詞題目生成器共用化 (`getSharedRandomPoem` 演算法)
- **歷史痛點 (Pain Points)**：
  在《花月》的各個遊戲中（如 Game 4、Game 7、Game 9 等），核心玩法多與「隨機抽取詩詞」有關。初期開發時，各個遊戲往往各自維護一套抽取邏輯，導致：
  1. 程式碼高度重複且難以統一維護。
  2. 抽出的詩句可能不連貫（例如抽到標點符號截斷或非偶數句結尾），破壞題目體驗。
  3. 各遊戲對題目的難度（字數、行數、評分）缺乏統一標準。
  4. 隨機性不佳，若依評分排序常導致玩家反覆玩到同一首高分詩詞。
- **優化實作方案 (Implementation)**：
  我們在 `script.js` 實作了統一的 `getSharedRandomPoem(minRating, minLines, maxLines, minChars, maxChars)` 方法供所有遊戲呼叫：
  1. **精準條件篩選**：允許各遊戲依據自身設計傳入最低評分 (`minRating`)、所需行數範圍 (`minLines`~`maxLines`) 與總字數限制 (`minChars`~`maxChars`)，演算法會確保抽出的內容完全符合關卡需求。
  2. **智慧連貫擷取**：
     - 演算法限制必須從「奇數句」(index 0, 2, 4...) 開始抽取，確保符合古典詩詞「兩句一聯」的對仗與語意完整性。
     - 若所需字數與行數允許，演算法會自動以「2 句為單位」往下連續向後擷取擴充，直至滿足條件。
  3. **自動淨化處理 (Auto-Cleaning)**：
     - 抽取的詩句會自動利用正則表達式 `replace(/[，。？！、：；「」『』\s]/g, "")` 過濾掉所有標點符號與空白，直接回傳適合做為遊戲題目判定用的乾淨字串陣列。
  4. **絕對隨機與降級保護 (Absolute Random & Fallback)**：
     - 全程禁止以 `rating` 作為陣列排序依據，改以隨機亂數 (`Math.random()`) 在符合條件的詩集中抽取，最大化題庫多樣性。
     - 若條件設定過苛導致無詩可用，系統內建「降級保護機制」，自動放寬評分要求重新篩選，防止遊戲卡死崩潰。
- **結論**：
  往後設計與詩詞相關的新遊戲時，不再需要煩惱字詞截斷或隨機品質問題，只需呼叫 `getSharedRandomPoem` 並套入該關卡的字數/行數需求，即可獲得格式完美、乾淨且連貫的題目陣列。

### 3.3 CSS 載入防護機制（`loadCSS` 模式）
- **歷史痛點**：
  CSS 是全域生效的，各個頁面引入的所有 `<link rel="stylesheet">` 都會互相疊加。一旦兩個模組使用相同的 class 名稱，就會相互干擾。例如 `cards.css` 的 `.poem-line:first-child` 規則污染了 `calendar.js` 的詩句渲染。
- **統一解法**：在每個 JS 模組的頂部或 `init()` 函式中加入以下 loadCSS 防護，確保：
  1. 各自的 CSS 檔確實被載入（即使 HTML 沒有明確引入）
  2. 利用唯一的 `id` 標記防止重複載入

  ```javascript
  // 在 JS 模組的 init() 或頂部立即執行
  if (!document.getElementById('moduleX-css')) {
      const link = document.createElement('link');
      link.id = 'moduleX-css';
      link.rel = 'stylesheet';
      link.href = 'moduleX.css';
      document.head.appendChild(link);
  }
  ```

- **已加入 loadCSS 防護的模組清單**（截至 2026-05）：

  | JS 檔案 | CSS id 標記 | 加入位置 | 狀態 |
  |---------|------------|--------|------|
  | `calendar.js` | `calendar-css` | 頂部 IIFE | ✅ |
  | `cards.js` | `cards-css` | 頂部 IIFE | ✅ |
  | `author_bio.js` | `author-bio-css` | `init()` 開頭 | ✅ |
  | `achievement.js` | `achievement-css` | `init()` 開頭 | ✅ |
  | `difficulty-selector.js` | `difficulty-selector-css` | `init()` 開頭 | ✅ |
  | `poem_dialog.js` | `poem-dialog-css` | `init()` 開頭 | ✅ |
  | `introCard.js` | `intro-card-css` | `init()` 開頭 | ✅ |
  | `game1.js`~`game4.js` | `game1-css`~`game4-css` | 各 `loadCSS()` 方法 | ✅ |
  | `game5.js`~`game7.js` | `game5-css`~`game7-css` | Canvas 遊戲，依賴 HTML `<link>` | ⚠️ 無動態防護 |
  | `game8.js`~`game13.js` | `game8-css`~`game13-css` | 各 `loadCSS()` 方法 | ✅ |
  | `game14.js` | `game14-css` | 直接依賴 HTML `<link>` | ⚠️ 無動態防護 |
  | `ruleNoteDialog.js` | — | CSS 已內聯於 HTML | ⚠️ 無動態防護 |
  | `cloudSaveDialog.js` | — | CSS 已內聯於 HTML | ⚠️ 無動態防護 |

- **結論**：新增任何模組時，必須同步加入 loadCSS 防護，並確保 CSS class 名稱有模組前綴，才能徹底避免全域污染。Canvas 遊戲（game5/6/7）與 game14 目前無動態 loadCSS 防護，若未來移到非 index.html 環境下需補上。

## 4. 新增遊戲應注意事項 (New Game Checklist)

每次新增遊戲（gameXX）時，**必須同步修改以下所有位置**，缺少任何一項都會引發靜默性 bug（介面無法顯示、導航失效、成就未記錄、分數計算錯誤等）。以 Game15「墨韻游龍」為例說明具體做法。

---

### 4.1 新遊戲自身的必要規範（`gameXX.js` / `gameXX.css`）

| 規範 | 說明 | 常犯錯誤 |
|------|------|---------|
| **CSS class 前綴** | 所有 class 名稱加 `gameXX-` 前綴（如 `game15-overlay`）；絕不使用通用名稱 | `.overlay`、`.header` 等通用名稱會污染全域 CSS |
| **`loadCSS()` 防護** | 在 `init()` 或模組頂部以唯一 `id` 動態載入 CSS，防止重複載入 | 忘記加入導致在非 index.html 環境 CSS 失效 |
| **`stopGame()` 必須隱藏 overlay** | 遊戲的 overlay 掛在 `document.body`（非 `#stage`），`menu.js` 的全域清理只呼叫 `stopGame()`，**不呼叫** `hide()`。因此 `stopGame()` 自身必須執行 `this.container.classList.add('hidden')`，否則切換頁面後 overlay 仍覆蓋 calendar/card/author_bio | 只在 `hide()` 裡隱藏，`stopGame()` 忘記隱藏，導致從遊戲切換到其他頁面時畫面被遮蔽 |
| **掛載至 `document.body`** | 使用 `document.body.appendChild(div)` 而非掛至 `#stage`；`#stage` 有 CSS `transform` 會造成 `position:fixed` 雙重縮放 | 掛至 `#stage` 會讓 overlay 在縮放後座標偏移至畫面右半 |
| **`registerOverlayResize` 同步縮放** | 在 `createDOM()` 中呼叫 `window.registerOverlayResize((r) => { ... })` 同步 left/top/scale | 忘記註冊後在非標準螢幕尺寸上 overlay 座標錯位 |
| **`window.GameXX = GameXX`** | 在模組末尾掛載至全域，讓 `menu.js` 可以呼叫 | 忘記掛載後 `menu.js` 呼叫 `window.Game15` 得到 `undefined`，遊戲無法啟動 |
| **URL 自動啟動** | 監聽 `DOMContentLoaded`，偵測 `?game=XX` 自動呼叫 `show()` | 忘記加入後關卡挑戰模式的直連 URL 無法運作 |

**`stopGame()` 最小正確範本：**
```javascript
stopGame: function () {
    this.isActive = false;
    this.stopGameLoop();
    // ⚠️ 必須在此隱藏：menu.js 全域清理只呼叫 stopGame()，不呼叫 hide()
    if (this.container) this.container.classList.add('hidden');
},
```
**時間倒數框** 紅色四方形計時條邊框(只能放在遊戲區四周)
**關卡挑戰** 必須正確儲存每一關的通關紀錄。

### 4.4 `scoreManager.js` — 需修改的一處位置

**`gameSettings` 物件（約第 23 行）**：
```javascript
gameSettings: {
    // ... 既有項目 ...
    'game15': { base: 100, heart: 10, time: 3, getPointA: 15 }
    //           ↑通關基礎分  ↑每顆紅心  ↑每秒剩餘時間  ↑遊戲中每次得分
},
```
參數說明：
- `base`：`playWinAnimation` 階段0給予的通關固定分（通常 100）
- `heart`：每顆剩餘紅心換算的分數（通常 10）
- `time`：每秒剩餘倒數換算的分數（依遊戲節奏調整，時限長的可低至 1）
- `getPointA`：遊戲中每次正確操作的即時得分（各遊戲差異大，需自行設定）
- `getPointB`（選填）：第二種即時得分（如擊石鳴詩的「消除一整行」獎勵）

> 漏掉後 `ScoreManager.gameSettings.game15` 為 `undefined`，`playWinAnimation` 會以預設值 base=100/heart=10/time=5 計算，且 `getPointA` 讀取會拋出 undefined 錯誤。
